Getting started

We start from the following callback hell: unit-hell

type Hom a b = a -> b
type Callback a = Hom a ()
type Hell a = Hom (Callback a) ()

hell_new :: a -> Hell a
hell_new a ax = ax a

hell_type_1 :: Hom a b -> Hell a -> Hell b
hell_type_1 ab axx bx = axx \a -> bx (ab a)

hell_type_2 :: Hom a (Hell b) -> Hell a -> Hell b
hell_type_2 abxx axx bx = axx \a -> abxx a bx

The way to create a hell is only one, i.e. hell_new. On the other hand, there are two ways to grow a hell, i.e. hell_type_1 and hell_type_2.

Hell has a significant feature that it always returns the meaning-less result (). The usefulness of Hell lies only in the side-effect. Hell is a naive simulation of setTimeout of JS.

Continuation monad

unit-hell is a special case of the continuation monad. Now we take the integration with respect to the parameter object of continuation monad instead of fixing it to meaning-less type. Js⁡A:=∫P∈Set[[A,P],P]≃Yoneda⁡A, where [A,B] is a set of maps from a set A to a set B, the integration symbol ∫P∈Set represents an end. The Yoneda lemma says Js is isomorphic to the identity monad. Js is well fitted to be represented in Haskell.

type Hom a b = a -> b
type Js a = forall p. Hom (Hom a p) p

js_fw :: a -> Js a
js_fw a ap = ap a

js_bw :: Js a -> a
js_bw ma = ma id

js_fw and js_bw are in the relation of inverse. Hence any element of Js a is written in the form ($ a). In terms of programming, any element of Js a is a function that remembers the value passed at construction time. The construction value can be obtained by applying js_bw.

Js is an extension of Hell in the following sense:

js_to_hell :: Js a -> Hell a
js_to_hell ma ax = ma ax

In other word, Hell is a special case of Js. Js is a something deformation of Hell in which the deformation parameter is a terminal callback, i.e. ab in the following code:

js_terminal_callback :: Js a -> Hom a b -> b
js_terminal_callback ma ab = ma ab

Js enable us to obtain the final value of event without breaking a callback chain.

We write a monad toolkit of Js as follows:

hom_snd = (.)
hom_fst = flip hom_snd

js_pure = js_fw
js_get = js_bw

js_fmap :: Hom a b -> Hom (Js a) (Js b)
-- js_fmap ab = hom_fst (hom_fst ab)
js_fmap ab ma bp = bp $ ab $ js_get ma

js_join :: Js (Js a) -> Js a
-- js_join = hom_fst js_pure
js_join = js_get

js_kext :: Hom a (Js b) -> Hom (Js a) (Js b)
-- js_kext amb = js_join . js_fmap amb
js_kext amb ma bp = bp $ js_get $ amb $ js_get ma

js_ap :: Js (Hom a b) -> Hom (Js a) (Js b)
-- js_ap mab ma = js_kext (\ab -> js_fmap ab ma) mab
js_ap mab ma bp = bp $ js_get mab $ js_get ma

js_fmap_2 :: (a -> b -> c) -> Js a -> Js b -> Js c
-- js_fmap_2 abc ma mb = js_pure abc %*% ma %*% mb where
-- (%*%) = js_ap
js_fmap_2 abc ma mb cp = cp $ abc (js_get ma) (js_get mb)

Let’s compare the composition of js_fmap and that of js_kext.

js_fmap_fmap :: Hom b c -> Hom a b -> Hom (Js a) (Js c)
-- js_fmap_fmap bc ab = js_fmap bc . js_fmap ab
js_fmap_fmap bc ab ma cx = cx $ bc $ ab $ js_get ma

js_kext_kext :: Hom b (Js c) -> Hom a (Js b) -> Hom (Js a) (Js c)
-- js_kext_kext bmc amb = js_kext bmc . js_kext amb
js_kext_kext bmc amb ma cx = cx $ js_get $ bmc $ js_get $ amb $ js_get ma

We visualize the code by putting js_fmap_fmap and js_kext_kext together into one diagram as follows: ∫P∈Set[[A,P],P]get∫P∈Set[[B,P],P]get∫P∈Set[[C,P],P]getAabambBbcbmcCcxX. The codes of js_fmap_fmap and js_kext_kext are different, but the spaces of them are the same because the isomorphism Hom a b == Hom a (Js b) holds. Any js_kext_kext is written in the form of js_fmap_fmap, and vise versa.

Implementation of Promise

We can customize js_get to obtain a value of the event as follows:

js_apply_get :: Hom a (Js b) -> Hom a b
-- js_apply_get amb a = js_get $ amb a
js_apply_get amb a = amb a hook where hook = id
The hook corresponds to (goto, fail) => ... in the constructor of Promise new Promise ((goto, fail) => ...). By using hook, we can enhance Js to cache values of the event, and by using these cached values, we can extend a callback chain even after finishing the tasks of existing callback chain. This functionality is only available for firing once type callback chains. Those are what Promise supports.

Let codes be able to write with do-notation.

newtype Ks a = Ks {
 runKs :: Js a
 }

ks_get :: Ks a -> a
ks_get (Ks ma) = js_get ma

instance (Eq a) => Eq (Ks a) where
 (==) :: Ks a -> Ks a -> Bool
 ma == mb = ks_get ma == ks_get mb

instance (Show a) => Show (Ks a) where
 show :: Ks a -> String
 show ma = "Ks " ++ show (ks_get ma)

instance Functor Ks where
 fmap :: Hom a b -> Hom (Ks a) (Ks b)
 fmap ab (Ks a) = Ks $ js_fmap ab a

instance Applicative Ks where
 pure :: a -> Ks a
 pure a = Ks $ js_pure a
 (<*>) :: Ks (Hom a b) -> Hom (Ks a) (Ks b)
 (<*>) = ap

instance Monad Ks where
 (>>=) :: Ks a -> Hom a (Ks b) -> Ks b
 Ks ma >>= amb = amb $ js_get ma

Since Ks is isomorphic to Identity, add-hs is reproduced as follows:

none :: Ks Int
none = do
 let add = (+)
 x <- pure 1
 let add_1 = add x
 x <- pure $ 10 ^ 3
 pure $ add_1 0
 
none -- Ks 1

Let’s calculate 2 * 2 + 2 * 3 == 10.

ks_ :: Hom a b -> Hom a (Ks b)
ks_ = hom_snd pure

none :: Ks (Hom Int Int) -> Ks Int
none ma = do
 let mb = ma >>= ks_ ($ 2)
 let mc = ma >>= ks_ ($ 3)
 liftA2 (+) mb mc
 
none $ pure (2 *) -- Ks 10

Here we use ks_ as a mimic of fork a thread now. This code is directly translated to Promise.

Combining with other functor

We calculate 2 * 2 + 2 * 3 == 10 again.

type Es = Either String

none :: Ks (Es (Hom Int Int)) -> Ks (Es Int)
none ma = do
 let mb = ma >>= ks_ (either Left throw)
 let mc = ma >>= ks_ (($ 3) <$>)
 liftA2 try mb mc
 --
 where
  try eb ec = either catch Right $ liftA2 (+) eb ec
  catch e = Left $ "caught: " ++ e
  throw a = if out < 0 then Right out else fail where
   fail = Left $ "fail because b = " ++ show out
   out = a 2
 
none $ pure $ pure (2 *) -- Ks Left "caught: fail because b = 4"

Memorandum

Commutativity

Js commutes with any endo-functor F. F∫P∈Set[[A,P],P]≃FA≃∫P∈Set[[FA,P],P] In general, the commutativity of two monads isn’t a sufficient condition so that the composition of them is also an monad. However, exceptionally, both MJs and Js⁡M are also monads for every monad M thanks to the fact that Js is essentially the identity monad, So we can define monad transformers for both MJs and Js⁡M.

The commutativity leads us to the following isomorphism: FnJsn≃(Js⁡F)n≃Jsn⁡Fn. I don’t have any idea so far to utilize this isomorphism.

co-Yoneda lemma

In the case of Promise, the Yoneda lemma has given the place holder to obtain the value of event, and it has helped us. How about the case of co-Yoneda lemma?

  • co-Yoneda lemma in nLab

Let F,G:A→Set be functors. By the following calculation: [A,Set](F,G)≃∫a∈A[Fa,Ga]≃∫a∈A[Fa,∫b∈A[[a,b],Gb]]≃∫a,b∈A[Fa,[[a,b],Gb]]≃∫a,b∈A[Fa⊗[a,b],Gb]≃∫b∈A[∫a∈AFa⊗[a,b],Gb], we have Fa≃∫b∈AFb⊗[b,a]for alla∈A. This is the co-Yoneda lemma.

The following diagram gives a bird view for Yoneda lemma and co-Yoneda lemma: [X,A(a,b)][1,F]Yonedaco-Yoneda[X⊗Fa,Fb]curry[X,[Fa,Fb]]flip[Fa,[X,Fb]]. This diagram also gives the half of isomorphism in an explicit form.

We put SjA:=∫X∈Set[X,A]⊗X.

data Sj a = forall x. Sj {
 _fst :: Hom x a
 , _snd :: x
 }

sj_fw :: a -> Sj a
sj_fw = Sj id

sj_bw :: Sj a -> a
sj_bw (Sj xa x) = xa x

In this case, the place holder sj_bw is essentially the evaluation map. Hence we have a chance to customize the evaluation map.
