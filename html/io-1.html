Getting started

Lets consider the following code: add-r

none = with (new.env (), {
 add <- \(x) \(y) x + y;
 a <- 1;
 add_1 <- add (a);
 a <- 1e+3;
 add_1 (0) |> print ();
});
[1] 1000

We expected to print 1, but printed 1000 in actual. This is unexpected and unpleasant behavior.

There are several workarounds to avoid the behavior of add-r.

none = with (new.env (), {
 add <- \(x) \(y) x + y;
 add_pass <- \(x) \(y, z = x) z + y;
 add_force <- \(x) {force (x); \(y) x + y};
 add_need <- \(x) {x ; \(y) x + y};

 x <- 1;

});

The way of add_force is safe for any custom made function returning function newly written. On the other hand, it is necessary to call with forceAndCall for some function returning function provided by library. However it is impossible to decide whether it is necessary or not without seeing the source code. So it is safe to call with forceAndCall for any function returning function provided by library . Furthermore the more simple rule is the easier to remember. Finally we conclude that:

  • calling every function returning function with forceAndCall.

Do-notation

Lets check similar codes to add-r in other programming languages. The first one is Haskell whose evaluation strategy being classified as call by need as R. add-hs

runIdentity do
 let add = (+)
 a <- pure $ 1
 let add_1 = add a
 a <- pure $ 1e+3
 pure $ add_1 0
-- 1

The second one is Python whose evaluation strategy being classified as strict. add-py

def none ():
 add = lambda x: lambda y: x + y;
 a = 1;
 add_1 = add (a);
 a = 1e+3;
 print (add_1 (0));
 return None;
none ();
1

Both add-hs and add-py work as expected. This fact suggests that the behavior of add-r isnt due to Rs evaluation strategy.

According to the rule of do-notation, add-hs is translated to:

infixl 1 %>%
(%>%) = flip id

runIdentity $
 (+) %>% \add ->
 pure 1 >>= \x -> 
 add x %>% \add_1 ->
 pure (10 ^ 3) >>= \x ->
 pure (add_1 0)
-- 1

This code is equivalent to the following:

(+) %>% \add ->
 1 %>% \x -> 
 add x %>% \add_1 ->
 10 ^ 3 %>% \x ->
 add_1 0
-- 1

By using the rule of do-notation ineligibly, we translate add-r as follows:

(\(add = \(x) \(y) x + y) {
 (\(x = 1) {
  (\(add_1 = add (x)) {
   (\(x = 1e+3) {
    add_1 (0) |> print ()
   }) ()
  }) ()
 }) ()
}) ()
[1] 1

Now this code works as same as Haskells or Pythons one. Of course, this code is evaluated by call by need too. lambda-r

The unpleasant behavior of add-r is frequently associated to the call by need. It is correct though, there is an another point of view, i.e.the behavior of add-r is due to the specification of assignment <-.

If we were eliminating all assignments <- from a code like lambda-r, the unpleasant behavior of add-r would not happen. There are two different statements against the unpleasant behavior of add-r:

  1. with the assumption of the specification of assignment <-, one says that the behavior of add-r due to the call by need,
  2. with the assumption of lambda calculus, one says that the behavior of add-r due to the specification of assignment <-.

In any case, there might be non-trivial assumptions.

Here we think that do-notation is an expression of the similarity between a Kleisli category and programming language. Since Kleisli dont know the evaluation strategy of each programming language, he/she gives a set of formulae that can be used in both evaluation strategies, strict and call by need. add-hs is an instance of special case of the identity monad, and the bind <- plays only the role of shadowing in Rust. If R were introducing the rule of shadowing in its specification, the tragedy of add-r would not happen. It seems that this is a kind of abstract nonsense, i.e.getting a solution by thinking in more general situation.

Example of function returning function

We consider the MASS::mvrnorm in this section. MASS::mvrnorm draws random numbers from the specified multivariate normal distribution.

  • MASS/mvrnorm.R at master cran/MASS GitHub

Lets imagine the situation that we want to simulate using random numbers drawn from some fixed multivariate normal distribution:

Sigma = ...
for (j in 1 : 1e+6) {
 x = MASS::mvrnorm (Sigma = Sigma);
 ...
}

This simulation will waste our time because this code repeats the diagonalization of matrix Sigma in the loop. So we want to separate the entire function into two parts, the part of diagonalization of matrix and the part of generating random numbers.

self = with (new.env (), {
 rnorm_for_loop = \(Sigma, negative = - 1e-3) {
  len = nrow (Sigma);
  if (len != ncol (Sigma)) {
   stop ("Sigma must be a square matrix.");

  eig = eigen (Sigma, symmetric = T);
  val = eig $ value;
  if (any (val < negative)) {
   stop ("Sigma must be positive-definite.");
  }
  val = pmax (sqrt (val), 0);
  mat = eig $ vectors;
  \() mat %*% (val * rnorm (len)) |> drop ();
 };

map_2 :: (Monad m) => Hom (a, b) c -> (m a, m b) -> m c
map_2 = uncurry . liftA2 . curry

Thermodynamics of programming

In thermodynamics of programming, int goes to double as the entropy increasing. Therefore the literal expression 123 is regarded as int (123) in most programming languages.

print (type (123));
<class 'int'>

However it isnt true in R.

typeof (123);
[1] "double"
This specification might be one of mysterious traditions of R.
