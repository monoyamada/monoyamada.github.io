Do-notation

According to the rule of do-notation, add-hs is translated to:

infixl 1 %>%
(%>%) = flip id

runIdentity $
 (+) %>% \add ->
 pure 1 >>= \x -> 
 add x %>% \add_1 ->
 pure (10 ^ 3) >>= \x ->
 pure (add_1 0)
-- 1

This code is equivalent to the following:

(+) %>% \add ->
 1 %>% \x -> 
 add x %>% \add_1 ->
 10 ^ 3 %>% \x ->
 add_1 0
-- 1

By using the rule of do-notation ineligibly, we translate add-r as follows:

The unpleasant behavior of add-r is frequently associated to the call by need. It is correct and impressive though, there is an another point of view, i.e. the behavior of add-r is due to the specification of assignment <-. If we can eliminate all assignments <- from a code like lambda-r, the unpleasant behavior of add-r will never happen. There are two different statements against the unpleasant behavior of add-r:

  1. with the assumption of the specification of assignment <-, one says that the behavior of add-r due to the call by need,
  2. with the assumption of lambda calculus, one says that the behavior of add-r due to the specification of assignment <-.

In any case, there might be non-trivial assumptions.

Here we think that do-notation is an expression of the similarity between a Kleisli category and programming language. Since Kleisli don’t know the evaluation strategy of each programming language, he/she gives a set of formulae that can be used in both evaluation strategies, strict and call by need. add-hs is an instance of special case of the identity monad, and the bind <- plays only the role of shadowing in Rust. If R were introducing the rule of shadowing in it’s specification, the tragedy of add-r would not happen. It seems that this is a kind of abstract nonsense, i.e. getting a solution by thinking in more general situation.

Promise of JS

In this section, we will compare Promise of JS and the identity monad.

We start from the following callback hell: unit-hell

type Hom a b = a -> b
type Callback a = Hom a ()
type Hell a = Hom (Callback a) ()

hell_new :: a -> Hell a
hell_new a ax = ax a

hell_add_type_1 :: Hom a b -> Hell a -> Hell b
hell_add_type_1 ab axx bx = axx \a -> bx (ab a)

hell_add_type_2 :: Hom a (Hell b) -> Hell a -> Hell b
hell_add_type_2 abxx axx bx = axx \a -> abxx a bx

Hell has two methods to extend a callback chain, i.e. hell_add_type_1 and hell_add_type_2. Hell has significant feature that it only returns the meaning-less result (). The usefulness of Hell lies only in the side-effect. Hell is a naive imitation of setTimeout of JS.

waiting

unit-hell is a special case of the continuation monad. Now we take the integration with respect to the parameter object of continuation monad instead of fixing it to meaning-less type. Js⁡A:=∫P∈Set[[A,P],P]≃Yoneda⁡A, where [A,B] is a set of maps from a set A to a set B, the integration symbol ∫P∈Set represents an end. The Yoneda lemma says Js is isomorphic to the identity monad in the sense of above equation. Js is well fitted to be represented in Haskell.

type Hom a b = a -> b
type Js a = forall p. Hom (Hom a p) p

js_fw :: a -> Js a
js_fw a ap = ap a

js_bw :: Js a -> a
js_bw ma = ma id

js_fw and js_bw are in the relation of inverse. Hence any element of Js a is written in the form ($ a). In the language of programming, any element of Js a is a function that remembers a value used at construction time. The value is able to obtain by applying js_bw.

We write a monad toolkit of Js as follows:

hom_snd = (.)
hom_fst = flip hom_snd

js_pure = js_fw
js_get = js_bw

js_fmap :: Hom a b -> Hom (Js a) (Js b)
-- js_fmap ab = hom_fst (hom_fst ab)
js_fmap ab ma bp = ma $ bp . ab

js_join :: Js (Js a) -> Js a
-- js_join = hom_fst js_pure
js_join mma ap = mma ($ ap)

js_kext :: Hom a (Js b) -> Hom (Js a) (Js b)
-- js_kext amb = js_join . js_fmap amb
js_kext amb ma = amb $ js_get ma

js_ap :: Js (Hom a b) -> Hom (Js a) (Js b)
-- js_ap mab ma = js_kext (\ab -> js_fmap ab ma) mab
js_ap mab ma bp = bp $ js_get mab $ js_get ma

js_fmap_2 :: (a -> b -> c) -> Js a -> Js b -> Js c
-- js_fmap_2 abc ma mb = js_pure abc %*% ma %*% mb where
-- (%*%) = js_ap
js_fmap_2 abc ma mb cp = cp $ abc (js_get ma) (js_get mb)

Both js_fmap and js_kext extends a callback chain though, the difference is illustrated as follows:

js_fmap_fmap :: Hom b c -> Hom a b -> Hom (Js a) (Js c)
-- js_fmap_fmap bc ab = js_fmap bc . js_fmap ab
js_fmap_fmap bc ab ma cp = ma $ cp . bc . ab

js_kext_kext :: Hom b (Js c) -> Hom a (Js b) -> Hom (Js a) (Js c)
-- js_kext_kext bmc amb = js_kext bmc . js_kext amb
js_kext_kext bmc amb ma = bmc $ js_get $ amb $ js_get ma

In js_fmap_fmap, the event is directly passed among callbacks ab and bc with transforming its value. On the other hand, in js_kext_kext, the event is passed among callback amb and bmc with taking it’s value by applying js_get. Note that the behaviors of js_fmap_fmap and js_kext_kext are different at a glance, but they are the same because the isomorphism Hom a b == Hom a (Js b) holds. Any chain of js_kext_kext can be written in the form js_fmap_fmap. However this illusion of difference makes sense when Js translate to asynchronous environment.

To write code with do-notation, we need some spells.

newtype Ks a = Ks {
 runKs :: Js a
 }

ks_get :: Ks a -> a
ks_get (Ks ma) = js_get ma

instance (Eq a) => Eq (Ks a) where
 (==) :: Ks a -> Ks a -> Bool
 ma == mb = ks_get ma == ks_get mb

instance (Show a) => Show (Ks a) where
 show :: Ks a -> String
 show ma = "Ks " ++ show (ks_get ma)

instance Functor Ks where
 fmap :: Hom a b -> Hom (Ks a) (Ks b)
 fmap ab (Ks a) = Ks $ js_fmap ab a

instance Applicative Ks where
 pure :: a -> Ks a
 pure a = Ks $ js_pure a
 (<*>) :: Ks (Hom a b) -> Hom (Ks a) (Ks b)
 (<*>) = ap

instance Monad Ks where
 (>>=) :: Ks a -> Hom a (Ks b) -> Ks b
 Ks ma >>= amb = amb $ js_get ma

Hence I think, with 51% confidence, the compositions Js⁡M and MJs are monads for all monad M. type Es = Either String

none :: Ks (Es (Hom Int Int)) -> Ks (Es Int)
none ma = do
 let mb = ma >>= ks_ (either Left throw)
