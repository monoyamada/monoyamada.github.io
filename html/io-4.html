Getting started

We start from the following function:

zip_just :: (a -> b -> c) -> [a] -> [b] -> [c]
zip_just abc as bs 
 | length as == length bs = [abc (as !! j) (bs !! j) | j <- js]
 | otherwise              = undefined
 where
  js = [0 .. (length as - 1)]

The behavior of zip_just is similar to mapply, but its domain is very limited. zip_just abc as bs works only when length as == length bs holds. In programming, IMHO, zip_just is the best from as a function in basic libraries because it helps us to preventing from bugs of index out of bounds. However, we consider to extend zip_just to be the total function.

Total function

The phrase total function depends on total domain. Actually, if we were assuming the type AB:=nNAnBn, zip_just would be a total function. Here, we used the phrase total function as a mean of a function whose domain is written by builtin types of ordinal programming languages.

There might be programming languages that we can easily define the type AB. However, in such a case, we cant perform naive currying to ABC, then it brings us the other difficulties.

zip_min

The following zip_min might be one of the easiest extensions of zip_just:

zip_min :: (a -> b -> c) -> [a] -> [b] -> [c]
zip_min abc as bs = [abc (as !! j) (bs !! j) | j <- js] where
 js = [0 .. (n - 1)]
 n = min (length as) (length bs)

Lets consider the following question: zip_min_assoc

  • Does zip_min preserve associativities?

As an example, we test for the concatenation of strings.

test_associativitiy :: (Monoid a, Eq a) => [a] -> [a] -> [a] -> Bool
test_associativitiy fa fb fc = lhs == rhs where
 lhs = (fa %*% fb) %*% fc
 rhs = fa %*% (fb %*% fc)
 (%*%) = zip_min (<>)

test_concatenation :: [String] -> [String] -> [String] -> Bool
test_concatenation = test_associativitiy

quickCheck test_concatenation

This test shows that zip_min (++) is associative, and hence, the answer to zip_min_assoc is yes for the case of (++).

Now wed like to answer zip_min_assoc for general cases.

zip_min can be rewritten as follows:

zip_reader :: (a -> b -> c) -> (p -> q -> r) -> (Hom x a, p) -> (Hom x b, q) -> (Hom x c, r)
zip_reader abc pqr ap bq = (liftA2 abc, pqr) %<% ap %<% bq where
 (%<%) = tom_to

zip_min :: (a -> b -> c) -> [a] -> [b] -> [c]
zip_min abc as bs = to_list $ zip_reader abc min (to_map as) (to_map bs) where
 to_map xs = ((!!) xs, length xs)
 to_list (ja, n) = [ja j | j <- [0 .. (n - 1)]]

liftA2 in zip_reader is the one of Reader monad.

to_map and to_list are inverse of each other, i.e.both to_map . to_list == id and to_list . to_map == id hold. By using this fact, test_associativitiy is rewritten as follows: test_zip_min

test_associativitiy :: (Monoid a, Eq a) => [a] -> [a] -> [a] -> Bool
test_associativitiy fa fb fc = lhs == rhs where
 lhs = to_list $ (to_map fa %*% to_map fb) %*% to_map fc
 rhs = to_list $ to_map fa %*% (to_map fb %*% to_map fc)
 (%*%) = zip_reader (<>) min
 to_map xs = ((!!) xs, length xs)
 to_list (ja, n) = [ja j | j <- [0 .. (n - 1)]]

test_concatenation :: [String] -> [String] -> [String] -> Bool
test_concatenation = test_associativitiy

quickCheck test_concatenation

Then the problem comes down to the associativity of zip_reader (<>) min. We know that zip_reader (<>) min is associative thanks to the following facts:

  1. liftA2 (<>) is associative.
  2. min is associative.
  3. If both f :: a -> a -> a and g :: x -> x -> x are associative, then the tuple (f, g) is associative under the element-wise application.

Note that, in the third statement above, element-wise means element-wise of a tuple but not of an array.

Finally, the unit of zip_min (<>) is given by the following:

zip_min_unit :: (Monoid a) => [a]
zip_min_unit = repeat mempty

Note that the unit of min is an infinite length array.

zip_lcm

We define zip_mod by extending zip_min as follows: zip_mod

zip_mod :: (Int -> Int -> Int) -> (a -> b -> c) -> [a] -> [b] -> [c]
zip_mod pqr abc as bs = to_list $ zip_reader abc pqr (to_map as) (to_map bs) where
 to_map xs = (\j -> xs !! mod j n, n) where n = length xs
 to_list (ja, n) = [ja j | j <- [0 .. (n - 1)]]

By using zip_mod, we define zip_lcm as follows:

zip_lcm = zip_mod lcm

Contrast to the previous section, to_map and to_list in zip_mod arent inverse each others. to_list . to_map == id still holds but to_map . to_list isnt the identity. to_map . to_list makes a function being cyclic as follows:

to_map_to_list :: (Hom Int a, Int) -> (Hom Int a, Int)
-- to_map_to_list = to_map . to_list where
to_map_to_list (ja, na) = (cyclic ja na, na) where
 -- cyclic ja na k = [ja j | j <- [0 .. (na - 1)]] !! mod k na
 cyclic ja na = ja . (`mod` na)
 to_map xs = (\j -> xs !! mod j n, n) where n = length xs
 to_list (ja, n) = [ja j | j <- [0 .. (n - 1)]]

We modify test_zip_min as follows:

mod_associativitiy :: (Monoid a, Eq a) => (Int -> Int -> Int) -> [a] -> [a] -> [a] -> Bool
mod_associativitiy pqr fa fb fc = lhs == rhs where
 lhs = to_list $ cyclic (to_map fa %*% to_map fb) %*% to_map fc
 rhs = to_list $ to_map fa %*% cyclic (to_map fb %*% to_map fc)
 (%*%) = zip_reader (<>) pqr
 to_map xs = (\j -> xs !! mod j n, n) where n = length xs
 to_list (ja, n) = [ja j | j <- [0 .. (n - 1)]]
 cyclic (f, n) = (f . (`mod` n), n)

mod_concatenation :: (Int -> Int -> Int) -> [String] -> [String] -> [String] -> Bool
mod_concatenation = mod_associativitiy

quickCheck $ mod_concatenation lcm

The main difference of this test from test_zip_min is the existence of cyclic. Whereas each cyclic in mod_associativitiy effectively becomes id thanks to the properties of lcm and to_map.

By taking account that the unit of lcm is 1, the unit of zip_lcm (<>) is given by the following:

zip_lcm_unit :: (Monoid a) => [a]
zip_lcm_unit = pure mempty

Recall that the unit of zip_min is an infinite length array. To make the difference clear, we arrange zip_min and zip_lcm in the form of Applicative.

newtype ZipMin a = ZipMin {
 runZipMin :: [a]
 } deriving (Eq, Show, Functor)

newtype ZipLcm a = ZipLcm {
 runZipLcm :: [a]
 } deriving (Eq, Show, Functor)

instance Applicative ZipMin where
 pure :: a -> ZipMin a
 pure a = ZipMin $ repeat a

 liftA2 :: (a -> b -> c) -> ZipMin a -> ZipMin b -> ZipMin c
 liftA2 abc (ZipMin a) (ZipMin b) = ZipMin $ zip_min abc a b

instance Applicative ZipLcm where
 pure :: a -> ZipLcm a
 pure a = ZipLcm [a]

 liftA2 :: (a -> b -> c) -> ZipLcm a -> ZipLcm b -> ZipLcm c
 liftA2 abc (ZipLcm a) (ZipLcm b) = ZipLcm $ zip_lcm abc a b

Comparing to List monad

The difference between liftA2 of List monad and zip_lcm is easy to understand in the form of list comprehension.

list_monad_case :: (a -> b -> c) -> [a] -> [b] -> [c]
-- list_monad_case = liftA2
list_monad_case abc as bs = [abc (to_map as j) (to_map bs k) | j <- js, k <- ks] where
 to_map = (!!)
 js = [0 .. (length as - 1)]
 ks = [0 .. (length bs - 1)]

zip_lcm_case :: (a -> b -> c) -> [a] -> [b] -> [c]
-- zip_lcm_case = zip_lcm
zip_lcm_case abc as bs = [abc (to_map as j) (to_map bs j) | j <- js] where
 to_map xs j = xs !! mod j (length xs)
 js = [0 .. (lcm (length as) (length bs) - 1)]

In list_monad_case, there are two indices within the list comprehension. These two indices run independently within the list comprehension. On the other hand, there is only one index within the list comprehension in zip_lcm_case. This single loop behavior of zip_lcm is similar to the one of liftA2 of Reader monad. By the definition of zip_mod, this similarity is obvious.

Relevance to mapply

zip_lcm works like that:

zip_lcm (,) [1 .. 2] ""     -- []
zip_lcm (,) [1 .. 2] "a"    -- [(1,'a'),(2,'a')]
zip_lcm (,) [1 .. 2] "ab"   -- [(1,'a'),(2,'b')]
zip_lcm (,) [1 .. 2] "abc"  -- [(1,'a'),(2,'b'),(1,'c'),(2,'a'),(1,'b'),(2,'c')]
zip_lcm (,) [1 .. 2] "abcd" -- [(1,'a'),(2,'b'),(1,'c'),(2,'d')]

We classify the behavior of zip_lcm by its arguments. Let na := length as and nb := length bs in zip_lcm abc as bs, the classification is the following:

  1. if min na nb == 0, then the output is the empty list.
  2. if na is a multiple of nb or vise versa, then the output is similar to mapply (abc, as, bs, SIMPLIFY = F).
  3. if na and nb are coprime, then output is similar to outer (abc, as, bs) but the order of elements of the output is different.

This correspondence is fairly similar to the definition of product. The following is a test whether (,) is a product or not:

product_fw :: (Hom x a, Hom x b) -> Hom x (a, b)
product_fw = uncurry $ liftA2 (,)

product_bw :: Hom x (a, b) -> (Hom x a, Hom x b)
product_bw = NonEmpty.unzip

test_bw_fw :: (Fun Int Int, Fun Int Int) -> Int -> Int -> Bool
test_bw_fw (Fun _ xa, Fun _ xb) x y = lhs == rhs where
 lhs = (ya x, yb y)
 (ya, yb) = product_bw $ product_fw (xa, xb)
 rhs = (xa x, xb y)

test_fw_bw :: Fun Int (Int, Int) -> Int -> Bool
test_fw_bw (Fun _ xab) x = lhs == rhs where
 lhs = (product_fw $ product_bw xab) x
 rhs = xab x

quickCheck test_bw_fw
quickCheck test_fw_bw

The correspondence between Tidy and Linq is given by rproduct_fw and product_bw as follows:

tidy_to_linq :: ([a], [b]) -> [(a, b)]
tidy_to_linq (as, bs) = to_list $ product_fw (to_map as, to_map bs) where
 to_map = (!!)
 to_list ja = map ja [0 .. (n - 1)]
 n = if na == nb then na else undefined
 na = length as
 nb = length bs

linq_to_tidy :: [(a, b)] -> ([a], [b])
linq_to_tidy abs = tom_to (to_list, to_list) $ product_bw $ to_map abs where
 to_map = (!!)
 to_list ja = map ja [0 .. (n - 1)]
 n = length abs

test_tidy_to_tidy :: [(Int, Int)] -> Bool
test_tidy_to_tidy zipped_ab = lhs ab == ab where
 lhs = linq_to_tidy . tidy_to_linq
 ab = NonEmpty.unzip zipped_ab

test_linq_to_linq :: [(Int, Int)] -> Bool
test_linq_to_linq ab = lhs ab == ab where
 lhs = tidy_to_linq . linq_to_tidy

quickCheck test_linq_to_linq

Usually we arent aware of the Reader monad though, it often exists in the behind of fundamental facts.

tidy_to_linq_is_zip_just :: [(Int, Int)] -> Bool

Going to the product

zip_min, zip_lcm, and additionally liftA2 of List monad are summarized as follows: (mll)zip_min[[[p]],A][[[q]],B][,1][,1][[[min(p,q)]],A][[[min(p,q)]],B]product_fw[[[min(p,q)]],AB],zip_lcm[[[p]],A][[[q]],B][modp,1][modq,1][[[lcm(p,q)]],A][[[lcm(p,q)]],B]product_fw[[[lcm(p,q)]],AB],List monad[X,A][Y,B][fst,1][snd,1][XY,A][XY,B]product_fw[XY,AB], where [[p]]={0,,p1N} be a set of zero-based indices, is an inclusion map. Are there a pattern?

Let (A,,i) be a cartesian monoidal category with

  1. the projections a1ab2b,
  2. the associator :(ab)ca(bc),

and F be a functor F:ASet.

We define the natural transformation as follows: (product-mu)[Fa,A][Fb,B][F(ab),AB]:=[Fa,A][Fb,B][F1,1][F2,1][F(ab),A][F(ab),B]product_fw[F(ab),AB] is looked like to reproduce the above summary (mll). In the case of List monad, by taking Set itself and 1Set for A and F, respectively, reproduces (mll). For the cases of zip_min and zip_lcm, the configuration of A and F are the followings:

zip_min

Let A be the following posetal category:

  1. an object is either a natural number or infinity .
  2. a morphism pq is defined iff pq.

The set of objects of A is N{}, and a morphism is the relation pq. The initial and terminal objects of A are 0 and , respectively.

The product and coproduct of p,qA are given by min(p,q) and max(p,q), respectively. We visualize this by pullback/pushout diagram as follows: min(p,q)qpmax(p,q)pullback / pushout. The unit object of product min is the terminal object .

We define the functor F:ASet as follows: F(pq):=[[p]][[q]]. A posetal morphism is mapped to an inclusion map.

zip_lcm

Let A be the posetal category of followings:

  1. an object is a natural number,
  2. a morphism pq iff q is a multiple of p.

The set of objects of A is N, and a morphism is the relation pqpNq. The initial and terminal objects of A are 1 and 0, respectively.

The product and coproduct of p,qA are given by gcd(p,q)qplcm(p,q)pullback / pushout. The counit object of coproduct lcm is the initial object 1.

We define the functor F:AopSet as follows: F(pq)op:=[[q]]modp[[p]]. Note that the roles of product and coproduct are inverted in the opposite category. Hence lcm is a product in Aop, and 1 is a unit object of lcm in Aop. A mapped morphism modp corresponds to the function to_map in zip_mod.

Preserving monoids

We show several properties so that (product-mu) preserves monoids.

The associativity of is ensured by the associativity of (): ([Fa,A][Fb,B])[Fc,C]1[F(ab),AB][Fc,C][F((ab)c),(AB)C][F1,][Fa,A]([Fb,B][Fc,C])1[Fa,A][F(bc),BC][F(a(bc)),A(BC)], where is the associators of () in Set.

product_fw makes the following diagram commuting: (dup)Bfop[B,X][B,Y]product_fw[f,1][f,1][B,XY][f,1]A[A,X][A,Y]product_fw[A,XY]. The rest part of diagram, i.e.the lower incomplete part, is the standard associativity diagram of product_fw. Since we already know that the standard associativity diagram of product_fw commutes, the only thing that we have to prove is the following diagram commuting: (F((ab)c)F((ab)c))F((ab)c)(F11F21)F2[FF]F(FaFb)Fc?(F(a(bc))F(a(bc)))F(a(bc))(F1F12)F22(FaFb)Fc. By checking each component of tensor product (), we know that this diagram commutes thanks to A being cartesian category, and hence, we can erase the question of =? in this diagram. Thus the proposition has been proven.

The following diagram commutes: I[Fa,A]const1snd|I[Fi,I][Fa,A][Fa,A][F(ia),IA][F2|i1,snd|I], where

  1. snd|I:IAA is the restriction of projection snd to IA. By this restriction, snd|I is invertible,
  2. 2|i:iaa is similar restriction to snd|I.

This diagram is similar to the unit diagrams of monoidal functor, and hence, it plays a role to ensure preserving the unit of monoid.

By using (dup), we have I[Fa,A]const1snd|Iconst1[Fi,I][Fa,A][F1,1][F2|i,1][F(12)|i1,1]1[Fa,I][Fa,A]product_fw[F(ia),I][F(ia),A]product_fw[F2|i1,1][F2|i1,1][Fa,A][Fa,IA][1,snd|I][F(ia),IA][F2|i1,1]. Thus the proof of proposition boils down to showing the left bottom diagram of this diagram commuting. By the unitality of product_fw and the invertibility of snd|I, we can show that the left bottom diagram of this diagram commutes. Thus the proposition has been proven.

Reader monad

Reader monad can be regard as follows:

  • composition of maps + fundamental tuple operations.

pure and join of Reader monad are given by the projection fst and duplication dup, respectively. [A,C]fst1[1,pure][AB,C]curry[A,[B,C]],[AA,B]curry[dup,1][A,[A,B]]join[A,B]. The other projection snd is usually used in the following from: [B,C]snd1[AB,C]curry[A,[B,C]]1Set(1B)[B,B]. By putting L:=(.), i:=pure, j:=(1X), they form the axiom of closed category.

In this note, we are interested in liftA2 of Reader monad. liftA2 of Reader monad is essentially dup as follows:

dup :: a -> (a, a)
dup a = (a, a)

product_fw :: (Hom x a, Hom x b) -> Hom x (a, b)
-- product_fw = uncurry $ liftA2 (,)
-- product_fw (xa, xb) = tom_to (xa, xb ). dup
product_fw = hom_fst dup . tom_to

We can understand that the zip nature of Reader monad comes from dup of tuple.

It is rarely said that the tuple is a monad. In R, pairlist is a monad. The following functors between the category of sets and its
product category forms an adjunction: Set()DupSetSet, where Dup(AfB):=AAffBB,()(AXfgBY):=AXfgBX. The monad toolkit of tuple monad is the following:

tuple_fmap :: Hom a b -> Hom (a, a) (b, b)
tuple_fmap = tom_to . dup

tuple_pure :: a -> (a, a)
tuple_pure = dup

tuple_join :: ((a, b), (c, d)) -> (a, d)
tuple_join = tom_to fst snd

Fibered product

The phrase fibered product2 could be rephrased as the column-wise product in terms of spreadsheet. For example, lets consider the following pullback: (column-wise){0,2}{0,2,4}fiber over0+{1,3}{1,3,5}fiber over112[[22]]mod2[[2]][[23]]mod2. Each fiber bundle over the base space [[2]] in this picture can be written in the form of spreadsheet as follows: [[22]]=2301,[[23]]=452301. The pullback of (column-wise) is obtained by taking the cartesian product of [[22]] and [[22]] column-wisely.

none = with (new.env (), {
