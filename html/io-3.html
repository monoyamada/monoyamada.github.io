Getting started

Let’s consider the following set of functions:

self = with (new.env (), {
 id = \(a) a;
 flip = \(abc) \(...) \(a) abc (a) (...);
 clip = flip (id);
 drip = clip ();
 curry_fw = \(abc) \(a) \(...) abc (a, ...);
 curry_bw = \(abc) \(a, ...) abc (a) (...);
 hom_snd = \(bc) \(ab) \(...) bc (ab (...));
 hom_fst = flip (hom_snd);
 const = \(a) \(...) a;
 swap = hom_snd (curry_bw) (hom_snd (flip) (curry_fw));
 fst = curry_bw (const);
 dup = \(abc) \(a) abc (a) (a);

 `%.%` = curry_bw (hom_snd);
 `%>%` = curry_bw (clip);
 `%<%` = curry_bw (curry_fw);

 as.list (rlang::current_env ());
});

The set of functions (id, hom_snd, const) gives the axiom of closed category, flip gives a cartesian structure, the pair of curry_fw and curry_bw gives adjunction to the monoidal category. By adding a data structure representing the disjoint union, the above set of functions provides a toolkit of category of sets.

curry_bw (id) corresponds to the evaluation map, and works like do.call.

none = with (self, {
 do.call (`+`, list (1, 2)) |> print ();
 curry_bw (id) (`+`, 1, 2) |> print ();
});

[1] 3

Both %>% and %<% work like the pipe |>.

none = with (self, {

 1 |> why (2) |> print ();
 (why %<% 1) (2) |> print ();
 why %<% 1 %<% 2 %>% drip |> print ();
 why %>% curry_fw %>% clip (1) %>% clip (2) |> print ();
});

%<% is not a trick but essentially the associator of monoidal category. [A⊗(B⊗C),D]→curry[A,[B⊗C,D]]→[1A,curry][A,[B,[C,D]]]↓[α,1D]def↓ϕ∥[A⊗B)⊗C,D]→curry[A⊗B,[C,D]]→curry[A,[B,[C,D]]], where [A,B] is a set of maps from a set A to a set B, 1X:X→X is the identity map on a set X, α is the associator with respect to the tensor product (⊗) of monoidal category. The tensor product (⊗) corresponds to pairlist of R. %<% is given by %<% := phi (id).

Let’s consider the associativity of addition (1+2)+3=1+(2+3). This equation is written in the point-free form as (+)∘((+)⊗id)=(+)∘(id⊗(+))∘α, where α is the associator. The associator is related to the associativity of bivariate functions.

Since an argument list of function is not a first-class citizen in R as same as almost the other programming languages, we can’t directly define a function implementing the associator. The partial application %<% can be viewed as an indirect implementation of the associator as described above.

Thermodynamics of programming

In a programming, int tends to change to double as the entropy increasing.

def none ():
 a = 123;
 b = a / 2;
 print (type (a), type (b), sep = " -> ");
 return None;
none ();
<class 'int'> -> <class 'float'>

Therefore the literal expression 123 is regarded as int (123) in most programming languages. However it isn’t true in R.

typeof (123);
[1] "double"
This specification might be one of mysterious traditions of R.

mapply

In this section, we use the convention of zero-based indexing.

Let’s consider the isomorphism An≃[range⁡(n),A], where An is a set of arrays of A with length n, range⁡(n):={0,…,n−1∈N} is a set of zero-based indices, the name borrowed from Python so to easy to remember. There is a degree of freedom to embed [range⁡(n),A] to [N,A]. An≃[range⁡(n),A]→degree of freedom[N,A] Here we fix this degree of freedom by mod as follows:

type Hom a b = a -> b

hom_snd = (.)
hom_fst = flip hom_snd

list_to_map :: [a] -> Hom Int a
list_to_map as j = as !! mod j (length as)

map_to_list :: Int -> Hom Int a -> [a]
map_to_list n ja = map ja [0 .. (n - 1)]

These maps enable us to bring the reader monad’s stuff to arrays. Here we bring liftA2 to arrays.

zip_like :: (Int -> Int -> Int) -> (a -> b -> c) -> [a] -> [b] -> [c]
zip_like len abc as bs = to_list $ liftA2 abc ja jb where
 ja = list_to_map as
 jb = list_to_map bs
 to_list = map_to_list $ len na nb
 na = length as
 nb = length bs

zip_min = zip_like min
zip_lcm = zip_like lcm

zip_lcm works like that:

zip_lcm (,) [1 .. 2] ""     -- []
zip_lcm (,) [1 .. 2] "a"    -- [(1,'a'),(2,'a')]
zip_lcm (,) [1 .. 2] "ab"   -- [(1,'a'),(2,'b')]
zip_lcm (,) [1 .. 2] "abc"  -- [(1,'a'),(2,'b'),(1,'c'),(2,'a'),(1,'b'),(2,'c')]
zip_lcm (,) [1 .. 2] "abcd" -- [(1,'a'),(2,'b'),(1,'c'),(2,'d')]

Now we have two functions list_monad::map_2 and zip_lcm these are the same type but works differently. The behavior of list_monad::map_2 is illustrated as follows:

for (j in 1 : length (as)) {
 for (k in 1 : length (bs)) {
  out.append (abc (as [j], bs [k]));
 }
}

on the other hand, the behavior of zip_min, instead of zip_lcm for simplicity, is illustrated as follows:

for (j in 1 : min (length (as), length (bs))) {
 out.append (abc (as [j], bs [j]));
}

This single loop behavior is similar to the one of reader-map_2. By the definition of zip_like, this similarity is obvious.

Monoidal functor

zip_min together with the unit map pure a = repeat a forms Applicative, i.e.  ZipList. In contrast, zip_lcm together with the unit map pure a = [a] forms Applicative. Both zip_min and zip_lcm are Applicative but their pure are different.

In slight general, if a pair (len, len_unit) is a monoid on the natural numbers, and len makes the following diagram commutes for all p,q,r∈N: range⁡((p⊙q)⊙r)mod(p⊙q)modpmodqrange⁡(p)range⁡(p⊙q)modpmodqrange⁡(q),a⊙b:=len(a,b), then, by defining the function pure by pure a := replicate len_unit a, the pair (zip_like len, pure) forms Applicative.

Many binary operations in R are seemed like according to the style of mappy, i.e. shorter lists are cycled1. In terms of this note, many binary operations in R are translated from scalar operations by using zip_lcm. In this case, the monoidal functor plays a role of providing a set of functions with unified look and feel.

Both zip_min and zip_lcm give the correspondence between a record of lists and a list of records as follows:

import collections;

def none ():
 ABC = collections.namedtuple ("ABC", ("a", "b", "c"));
 tidy = ABC (tuple (range (3)), tuple ("abc"), (True, False, True));

 print (tidy, list (linq), sep = "\n");

none ();
ABC(a=(0, 1, 2), b=('a', 'b', 'c'), c=(True, False, True))
[ABC(a=0, b='a', c=True), ABC(a=1, b='b', c=False), ABC(a=2, b='c', c=True)]

This correspondence is related to the product in a category of sets.

product_fw :: (Hom x a, Hom x b) -> Hom x (a, b)
product_fw = uncurry $ liftA2 (,)

product_bw :: Hom x (a, b) -> (Hom x a, Hom x b)
product_bw = NonEmpty.unzip

tidy_to_linq :: ([a], [b]) -> [(a, b)]
-- tidy_to_linq (as, bs) = to_list $ product_fw (xa, xb) where
tidy_to_linq (as, bs) = zip_min (,) as bs where
 xa = list_to_map as
 xb = list_to_map bs
 to_list = map_to_list n
 n = min (length as) (length bs)

linq_to_tidy :: [(a, b)] -> ([a], [b])
linq_to_tidy abs = to_list $ product_bw xab where
 xab = list_to_map abs
 to_list (xa, xb) = (map_to_list n xa, map_to_list n xb)
 n = length abs
